Remote Worker Node in Day2 spoke cluster to add worker nodes

Requirements

The starting point of this document is based on the following things installed previously:

  • Hub clusters ocp ipi baremetal installed (multinode)
  • spoke cluster multinode installed. You could use the deploy.sh script to accomplish this.

Remote worker node into Day2 spoke cluster workflow

1-. Deploy hub cluster (Requirement)

2-. Add bmh crs to hub clusters (in order to deploy spoke cluster multi node) (Requirement)

3-. Finish the spoke cluster installation (multinode 3 nodes)

4-. Create the new VM into a different L3 network

5-. Add new manifest to create a new bmh on hub cluster (in this case with the worker role)

6-. Automatically cluster will be cluster-day2 and the worker will be added to the spoke cluster as a worker.

7-. Download kubeconfig for spoke-cluster

8-. Validate oc get nodes to have 3 nodes master and the new worker node in a different L3 network

Remember that the agentclusterinstall manifest when you’re using a multinode spoke cluster should be similar to:

apiVersion: extensions.hive.openshift.io/v1beta1
kind: AgentClusterInstall
metadata:
  name: lab-cluster-aci
  namespace: assisted-installer
spec:
  apiVIP: 192.168.150.243                  # replace with your value
  ingressVIP: 192.168.150.242              # replace with your value
  clusterDeploymentRef:
    name: lab-cluster
  imageSetRef:
    name: openshift-v4.8.0
  networking:
    clusterNetwork:
      - cidr: 10.128.0.0/14
        hostPrefix: 23
    serviceNetwork:
      - 172.30.0.0/16
  provisionRequirements:
    controlPlaneAgents: NUMBER_OF_MASTERS   # replace with your value
  sshPublicKey: PUB_SSH_KEY                 # replace with your value

As you can see, with 3 nodes you don’t need to specify the MachineNetwork.

New worker node to be added to the cluster

Once we have the spoke cluster deployed, the cluster will be a day2 cluster

  state: adding-hosts
      stateInfo: Cluster is installed

It does means that the cluster is ready to be used accepting new worker nodes.

  • Create a new virtual machine with the following parameters:
  kcli create vm -P uefi_legacy=true -P start=false -P nets=['{"name":"default", "mac":"ee:bb:aa:ee:1e:1f"}'] -P memory=38000 -P numcpus=8 -P disks=[280] agent4

In this case the vm will be in a different L3 network. Just to make easy:

  • Network Bare-net: 192.168.150.0/24 In this network will be Hub and Spoke cluster nodes/vm
  • Network Default: 192.168.122.0/24 In this network will be the remote worker node

So with this info we will have something like this. The agent7 will be the remote worker node in a different L3 Network as you can see here:

The VM agent 4 at the beginning is stopped and then with metal3 will be started once the bmh is created.

  • Creation of new baremetal host with the following parameters. One of the parameters is the role which in this case will be worker:
---
apiVersion: v1
kind: Secret
metadata:
  name: bmc-secret4
  namespace: assisted-installer
data:
  password: ---redacted---
  username: ---redacted---
type: Opaque


---
apiVersion: metal3.io/v1alpha1
kind: BareMetalHost
metadata:
  name: lab-agent4
  namespace: assisted-installer
  labels:
    infraenvs.agent-install.openshift.io: "lab-env"
  annotations:
    inspect.metal3.io: disabled
    bmac.agent-install.openshift.io/role: worker
    bmac.agent-install.openshift.io/hostname: "spoke4"
spec:
  online: true
  bmc:
    address: redfish-virtualmedia+http://192.168.150.149:8000/redfish/v1/Systems/2a93e59f-2d0b-47e7-a103-24e97ccb2639
    credentialsName: bmc-secret4
    disableCertificateVerification: true
  bootMACAddress: ee:bb:aa:ee:1e:1f
  automatedCleaningMode: disabled

As usual, the ID of the new VM is: 2a93e59f-2d0b-47e7-a103-24e97ccb2639 and the Mac Address: ee:bb:aa:ee:1e:1f

Downloading Kubeconfig from the spoke cluster

$ oc get secret lab-cluster-admin-kubeconfig -o jsonpath='{.data.kubeconfig}' | base64 -d > kubeconfig
$ export KUBECONFIG=kubeconfig

Now you can use the oc commands to interact with the spoke cluster.
