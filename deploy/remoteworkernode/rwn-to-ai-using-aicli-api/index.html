Scenario Description

Once we’ve got a 3 master cluster installed with the AI operator, this document is useful to add new remote worker nodes (maybe worker nodes using the same network)

In this case, we’ve got 2 possibilities to do that:

  • Using the API or maybe an abstraction using AICLI
  • Using the CRD operator (not developed yet)

Requirement

We need an environment with:

  • Hub deployment ready (UPI,IPI whatever)
  • AI operator + Hive + LocalStorage
  • Cluster 3 multi-node (not possible to do using SNO). The word single node is just to have one node without any possibility to add workers
[root@qct-d14u11 rwn-ai-crd]# oc get agents

Now, we’re going to deploy a remote worker node but in a different network L3 segment: assisted-net -> 192.168.7.0/24

Create new cluster *-day2 using API/library:

The idea is to create a new cluster adding a suffix ‚Äú-day2‚Äù in order to add the remote worker node to this cluster. The idea is basically join or create a trick to associate the new cluster -day2 to the original to add new hosts but not directly using the original cluster. So the steps should be:

  • Create a new cluster -day2
  • Create the discovery ISO associated to this new cluster -day2
  • Spin up a new VM (remote worker) using a different L3 network (assisted-net) booting from this Discovery-day2 image
  • Change the information required (hostname, api_vip, and so on)
  • FIX the L2 Support Environment changing the config map

Using AICLI:

export AI_URL=http://<url-api>:<port>
aicli create cluster cluster-rwn-alb2-day2

To verify day2 you could do the next command to view “adding-host” state:

Create Remote Worker VM

The most important thing here is that you have two possibilities:

  • Create a VM in the same network (Local worker)
  • Create a VM in another network (remote worker)

In our case we’re gonna use the remote worker node so, we’re gonna create the VM in another L3 network:

assisted network.

In this scenario:

  • Default network: 192.168.122.0/24
  • Assisted network: 192.168.7.0/24
[root@qct-d14u11 assisted-installer-cli]# kcli create vm -P iso=new-discovery-rwn.iso -P memory=10000 -P numcpus=8 -P disks=[120] -P nets=['{"name":"assisted-net"}'] remote-l3-new1

Update some node attributes:

It’s possible that you have to update some node’s attributes like hostname, api_vip and so on:

[root@qct-d14u11 ~]# aicli update host localhost -P role=worker -P approved=true -P name=remote-l3-new1 -P api_vip=192.168.122.183

Just to approve the CSR and start the installation successfully

TROUBLESHOOTING

Verify SUPPORT L2 environment:

If your new status host (RWN L3 VM) is ‚Äúinsufficient‚Äù, it could be due to the L2 verification because, for sure, if you’re getting your VM in another L3 network, you’re in another different broadcast domain so the check (https://github.com/openshift/assisted-installer-agent/blob/30cdb974bb9f7e5dc1ae5bfe03ad6ed7cd071df0/src/apivip_check/apivip_check.go#L119) fails. To verify that check that the validations:

  • Agent (192.168.7.186) and the API.VIP cluster AI (192.168.122.183)
  • The agent has connectivity to API VIP cluster
[core@localhost ~]$ ping api.cluster-rwn-alb2.qct11.karmalabs.com
